## Context

当前 BPM 设计器页面采用左右分栏布局，右侧 `bpm-properties-panel` 宽度固定。当属性项较多时，面板内容高度超过可视区域，现有容器缺少明确的高度和滚动约束，导致内容被裁切或布局错位。该问题集中在前端布局与样式层，不涉及后端数据结构。

## Goals / Non-Goals

**Goals:**
- 在属性项较多时，保证右侧属性面板可以纵向滚动并完整访问全部配置项。
- 保持设计器画布区与属性面板区稳定分栏，不因内容增长破坏整体布局。
- 以最小改动修复现有页面，避免影响 BPM 其他功能。

**Non-Goals:**
- 不重构属性面板业务逻辑与字段渲染机制。
- 不调整后端接口、流程定义模型或权限逻辑。
- 不引入新的 UI 组件库或第三方依赖。

## Decisions

1. 为编辑器容器建立“可伸缩 + 可滚动”布局约束  
   - 采用对 `nz-layout`/`nz-sider` 容器补充高度与 `min-height: 0` 约束的方式，使子容器在 flex 布局中可正确收缩。  
   - 备选方案是仅给属性面板组件加固定高度，但会在窗口尺寸变化或父级高度变化时失效，因此不采用。

2. 将滚动责任放在属性面板容器层，而不是全页容器  
   - 通过为右侧面板区域设置 `overflow-y: auto`（或等效容器滚动）确保只滚动属性区。  
   - 备选方案是页面整体滚动，但会影响左侧画布操作体验并引发工具栏位置不稳定，因此不采用。

3. 保持模板结构基本不变，仅增加语义化 class 以承载样式  
   - 在 `bpm-editor.html` 保留现有组件组织，新增必要 class 标识，便于精准控制布局。  
   - 备选方案是改写为更复杂的嵌套结构，改动面更大且收益有限，因此不采用。

## Risks / Trade-offs

- [风险] 不同浏览器对 `height: 100%` 与 flex 子项收缩行为存在差异 -> [缓解] 同时使用 `min-height: 0` 与局部滚动容器，减少兼容性问题。  
- [风险] 若 `bpm-properties-panel` 内部组件已有滚动逻辑，可能出现双滚动条 -> [缓解] 优先在外层容器统一滚动，并在实现时检查内部样式冲突。  
- [权衡] 固定右侧宽度可保持稳定编辑体验，但在窄屏场景下画布空间更紧张 -> [缓解] 本次先保证可用性，响应式优化留待后续独立变更。
